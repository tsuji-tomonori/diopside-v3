## 設計方針
- 本文書は、[[RQ-GL-001|diopside]] の画面を「利用者向け探索導線」と「管理者向け運用導線」に分けて定義し、要求から設計への対応を明確にする。
- 画面ごとに目的・入口・主要操作・状態を定義し、[[RQ-SH-002|利用者]] と [[RQ-SH-001|管理者]] が迷わず次操作へ進める情報設計を優先する。
- 画面一覧は見た目の列挙ではなく、[[RQ-UC-001]]〜[[RQ-UC-009]] の利用シナリオを成立させる最小単位として管理する。

## 設計要点
- 利用者導線は「一覧表示（[[RQ-UC-002]]）→絞り込み/検索（[[RQ-UC-003]]/[[RQ-UC-004]]/[[RQ-UC-005]]）→詳細確認（[[RQ-UC-006]]）」を基準とする。
- 管理者導線は「収集起動（[[RQ-UC-001]]）→監視/失敗調査（[[RQ-UC-007]]）→[[RQ-GL-011|再収集]]（[[RQ-UC-008]]）→配信前後再確認/手動タグ付け（[[RQ-UC-009]]）」を基準とする。
- 更新系UIはすべてBackend API経由でDB正本を更新し、利用者向け参照契約と直接結合しない。
- 詳細モーダル内の補助表示（[[RQ-GL-016|コメント密度波形]]/[[RQ-GL-017|ワードクラウド]]）は、主導線を阻害しない従属要素として配置する（[[BD-ADR-009]]/[[BD-ADR-010]]）。

## 更新系/参照系のUI境界
- 管理画面（UI-A01〜UI-A06）は更新系UIとしてBackend APIを必須経路にする。
- 利用者画面（UI-U01〜UI-U03）は参照系UIとして静的配信JSON/静的画像のみを参照する。
- 将来の高度検索は利用者画面に検索APIを追加可能とするが、管理画面導線は変更しない。

## 補足
- 画面カタログは「どの画面が存在するか」だけでなく、「どの要求をどの操作で受けるか」を示す索引として扱う。
- 具体レイアウト、部品単位の表示仕様、例外メッセージ文言は [[BD-UI-002]] と [[BD-UI-004]]、および [[DD-UI-001]]〜[[DD-UI-006]] で詳細化する。
