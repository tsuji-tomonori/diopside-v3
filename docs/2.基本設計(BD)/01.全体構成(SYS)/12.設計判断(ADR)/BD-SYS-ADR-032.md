## 決定事項
- Analytics（分析）を Supporting ではなく Core サブドメインとして扱う。
- Analytics は BC6 として恒久運用し、コンテキスト一覧の「オプション」扱いを廃止する。
- `[[BD-SYS-DOM-001]]` のサブドメイン分類定義に、Core/Supporting の判定基準を明記する。
- 境界契約マトリクスとBC責務サマリで、略語定義および日本語併記を標準化する。

## 理由
- [[RQ-FR-022]] と [[RQ-FR-023]] は、閲覧体験での理解性向上と再生導線改善を直接成立させる機能であり、補助機能として扱うと価値定義と不整合になる。
- [[RQ-GL-016]] と [[RQ-GL-017]] は Viewing の体験品質を規定する主要要素であり、差別化要素として継続投資対象に含める必要がある。
- 設計レビュー指摘で、分類定義不足・略語定義不足・日本語併記不足が検出されており、同一変更で解消する必要がある。

## 影響
- `[[BD-SYS-DOM-001]]` の Analytics 行は `Core / 最優先` に更新し、理由を価値直結の表現へ変更する。
- コンテキスト一覧の `BC6 Analytics（オプション）` は `BC6 Analytics` へ更新する。
- 境界契約パターン定義を追加し、上流/下流の英語表記には日本語を併記する。
- BC責務サマリの凡例として `BC = Bounded Context（境界づけられたコンテキスト）` を追加する。

## トレードオフ
- Analytics を Core 化することで、品質保証・変更管理・レビュー対象の優先度が上がり、運用コストは増加する。
- 一方で、[[RQ-SH-002|利用者]]価値に直結する分析機能の品質劣化を早期検出でき、価値毀損リスクを低減できる。

## 却下した選択肢
- Analytics を Supporting のまま据え置く案: 閲覧体験価値との整合が取れず、分類基準の一貫性を満たせないため不採用。
- Analytics を Core へ変更するが BC6 をオプション扱いのまま残す案: 分類と運用優先度が矛盾するため不採用。
