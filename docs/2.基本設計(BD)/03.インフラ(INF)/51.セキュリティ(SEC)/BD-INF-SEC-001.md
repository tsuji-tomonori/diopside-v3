## 方針
- 公開データのみを扱う前提を維持しつつ、公開配信境界と管理系境界を分離する。
- 更新系の信頼境界は `管理画面 -> Backend API -> DB` に固定し、機密情報を公開経路へ混在させない。
- ロールベースで最小権限を適用し、共有資格情報を禁止する。

## 認証・認可境界
- `'/openapi/*'` と `'/api/v1/*'` は認証必須とし、未認証・未認可アクセスで 2xx 応答を返さない。
- 管理系操作（[[RQ-GL-002|収集実行]]、[[RQ-GL-011|再収集]]、タグ更新、公開反映）は権限検証を通過した要求のみ実行可能とする。

## IAM・Secrets・鍵管理
- 秘密情報はSecrets管理サービスで保管し、平文保管を禁止する。
- `NEXT_PUBLIC_` を付与する設定は公開影響レビューを必須とし、機密値には付与しない。
- 機密情報（APIキー、トークン、接続情報）はサーバー側境界でのみ参照し、配信成果物へ含めない。

## 運用ルール
- 特権操作は期限付き昇格を採用し、監査ログを必須化する。
- 監査ログは `correlation_id` / `trace_id` を含む構造化ログで記録し、セキュリティイベント追跡を可能にする。

## 外部サービス評価
- 新規導入するサードパーティ/クラウドサービスは、導入前にアクセシビリティ適合状況（VPAT等）の確認記録を残す。
- 重大な欠陥が判明した場合は、機能停止/制限の判定基準と代替導線を定義する。
