2. 要求の矛盾分析

矛盾 #1: 継続収集 vs コスト上限・常時監視なし Medium

RQ-PP-001 / RQ-PC-006

RQ-PP-001 はプロジェクト目的として「継続的に収集・蓄積」を定義しているが、RQ-PC-006 は AWS 月額コストを 3,000 円に上限設定し、収集頻度の削減を優先事項としている。

影響: 「継続収集」の定義が曖昧なまま実装に進むと、コスト超過リスクまたは収集漏れのどちらかが発生する。

推奨: RQ-PP-001 に「継続収集」の定量的定義（例: 日次バッチ、週次バッチ）を追記し、RQ-PC-006 との整合性を明示する。

矛盾 #2: 出演条件判定 vs 手動確認 Low

RQ-FR-003 / RQ-SC-001

RQ-FR-003 は「出演条件一致の公開動画を判定して取り込める」と記述し自動判定を示唆する。一方 RQ-SC-001 では管理者による手動確認プロセスが前提。

影響: 完全自動化と手動確認付きワークフローのどちらを実装すべきか判断できない。

推奨: RQ-FR-003 に「管理者の確認を経て取り込む」フローを明記し、自動判定は候補提示にとどまることを明確化する。

矛盾 #3: 段階ロード無効化タイミング vs フィルタ即時動作 Medium

RQ-UC-002 / RQ-FR-009

RQ-UC-002 は段階ロードを定義し全データロード完了までフル絞り込みが無効。RQ-FR-009 はタグ条件による絞り込みが tag_master を参照して動作すると記述しており、archive_index ロード完了前のフィルタ動作が未定義。

影響: tag_master ロード完了後〜archive_index ロード完了前の間にタグフィルタ UI を有効化すべきかどうかが不明。

推奨: RQ-UC-002 に段階ごとの UI 状態遷移を明記する。

矛盾 #4: 脆弱性即時対応 vs 運用体制制約 Medium

RQ-SEC-001 / RQ-PC-004

RQ-SEC-001 はリリース時に Critical/High 脆弱性ゼロを受け入れ基準としているが、運用中の脆弱性対応時間は未定義。RQ-PC-004 は個人開発で 24 時間有人対応がないことを制約として明示。

影響: 運用中に Critical 脆弱性が発見された場合の対応 SLA が不明確。

推奨: RQ-SEC-001 に運用中の脆弱性対応基準を追記する（例: Critical 認知後 24h 以内に緩和策適用）。

3.3 LLM 統合の未反映

RQ-UC-009 / RQ-FR-005

RQ-UC-009 は管理者がタグ付けを行う際に「外部 LLM によるタグ提案」を利用するフローを定義。しかし RQ-FR-005（タグ辞書更新）には LLM 連携に関する記述がなく、機能要求レベルで LLM 統合が反映されていない。

推奨: RQ-FR-005 に LLM タグ提案機能の要求を追加するか、LLM 連携は運用手順であり機能要求対象外であることを明記する。

3.4 状態遷移図の欠如

  • 段階ロードの状態遷移: bootstrap → tag_master → archive_index の各段階での UI 状態・フィルタ有効範囲が図示されていない
  • タグライフサイクル: タグの作成 → 有効 → 無効化の遷移が DB 設計には暗示されているが状態遷移図がない

4. 基本設計の矛盾分析

BD 内部整合性: 良好（矛盾なし） Good

基本設計文書群内部で論理的矛盾は検出されなかった。

  • アーキテクチャ概要と個別設計文書の用語・構成が一致
  • データ設計と API 設計のエンティティ定義が整合
  • インフラ設計と品質特性の SLA 目標が整合

BD-DD アライメント: 良好 Good

詳細設計は基本設計を正しく拡張・具体化している。

  • BD-APP-API → DD-APP-API: API 仕様の詳細化
  • BD-INF-IAC → DD-INF-IAC: CI/CD パイプラインの具体化
  • BD-APP-DATA → DD-APP-DB: 物理設計への落とし込み

6. インフラ重点分析

6.1 Critical 欠落

RDS リストア手順の DD レベル詳細 Critical

BD-INF-DEP-006（方針定義済み） / DD 該当文書なし

BD-INF-DEP-006 で PITR・日次スナップショットの方針は定義されている（S3 バージョニング 30 日、DB 日次スナップショット + PITR 30 日、RTO/RPO 目標あり）。不足しているのは DD レベルでの具体的なリストアコマンド・手順書。

6.3 Minor 欠落

項目内容

6.4 良好な点

  • ネットワーク設計: CIDR/SG/NACL が実装可能レベルで具体的
  • 監視閾値: CloudFront 5xx（WARN 2% / CRIT 5%）、Lambda p95（WARN 1200ms / CRIT 1500ms）
  • IaC パイプライン: lint/validate → policy → cdk synth → cdk diff → approve → cdk deploy → smoke
  • 破壊的変更: 設計レビュー + 運用の二重承認を要求
