## テスト目的
- [[DD-APP-API-002]]〜[[DD-APP-API-015]] の契約が、実行基盤・配信データ・Web表示・運用導線で一貫して成立することを検証する。

## 観点
- API契約の入出力と、実データ連携結果が一致する。
- データ連携順序（[[RQ-GL-007|bootstrap]]→[[RQ-GL-008|タグマスター]]→archive）を検証する。
- 障害時の[[RQ-GL-011|再収集]]と診断導線を検証する。

## 結合テスト設計（境界・契約・連携順序）
- 契約境界は HTTP API を Hono実装から生成した OpenAPI（`/openapi/v1/openapi.json`）正本、静的JSON契約を JSON Schema 正本として扱い、DD-APP-API-* はフロー/制約/オラクルの設計入力として扱う。
- ITでは正本OpenAPI（リポジトリ生成物）と実装OpenAPI（起動時公開）を比較し、仕様差分をATへ持ち込まない。
- 連携順序は `bootstrap -> tag_master -> archive` の依存順を固定し、前段Fail時は後段ケースを停止する。
- 障害系は [[DD-APP-API-008]] / [[DD-APP-API-009]] を起点に、再実行・診断・復旧可否までを1セットで判定する。
- 配信経路と認証境界は [[IT-CASE-010]] と [[AT-SCN-006]] の接続前提を維持し、境界崩れの早期検出をITで担保する。

## 結合テスト仕様書フォーマット（UC基準）
- ITケースは API単位ではなく UC単位の利用シナリオを基準に設計し、ケース本文に `対象UC` と `生成元PWモデル` を必須記載する。
- 各ケースは「契約化した受入条件」「因子（機能要件ベース）」「オラクル判定」を持ち、要求から期待結果への判定経路を固定する。
- 因子は `RQ-FR-*` を主因子とし、APIパラメータ・データ状態・外部依存を補助因子として扱う。
- 組合せ強度は 2-wise を標準とし、冪等性・同時実行制約・障害復旧導線は 3-way 補完を許容する。
- 異常系は 1ケース1異常要因を原則とし、エラー優先順位のマスキングを回避する。

## UCベースPW構成
- PWモデルは `docs/5.結合テスト(IT)/12.ペアワイズ(PW)` を正本とし、対象は `RQ-UC-001` から `RQ-UC-009` の9本とする。
- `RQ-UC-010` から `RQ-UC-013`（CI/CD運用UC）は IT-PW 対象外とし、INFIT/AT運用系で検証する。
- ITケースは対応UCのPWモデルを参照し、契約観点・手順観点で必要ケースを抽出する。
